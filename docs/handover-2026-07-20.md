## 변경 요약

지난 세션(`docs/handover-2026-07-16.md`)에 남아있던 미커밋 SQL 수정 2건을 정리해서 커밋+push했고, 영상 등록 폼이 대회 목록을 불러올 때 불필요하게 무거운 API를 호출하던 비효율을 찾아 고쳤습니다. 이어서 사용자가 가져온 실제 대회(2026 사회인대회) 대진표 사진을 검토한 결과, 3인 1조 리그전 + 4개 조(A/B/C/D) 구조가 현재 대진표 스키마(1:1 매치 전용, 2개 조+결승만 지원)로는 표현이 안 된다는 걸 확인했습니다. 스키마/코드는 건드리지 않고, 기존 `division` 필드를 활용해 AB풀/CD풀/결선 3개로 쪼개는 방식으로 우회했습니다. **코드 변경분은 전부 커밋+push 완료**이며, 대진표 데이터(31개 매치 뼈대)는 **코드가 아니라 프로덕션 Supabase에 스크립트로 직접 삽입**해서 git에는 흔적이 없습니다.

## 대진표 데이터 작업 (git에 안 남는 프로덕션 변경)

코드 수정 없이, `.env.local`의 anon key로 Supabase `bracket_matches` 테이블에 스크립트로 직접 31개 행을 삽입했습니다. 대상: `사회인대회`(competition id `efe72ce1-94e2-4f4c-afd1-996ad68260a9`), 종목 "여자 청년부 개인전".

### 왜 스키마를 안 건드렸는지
- 사용자가 준 실제 대진표 사진은 **1라운드가 3인 1조 리그전**이라, `player1`/`player2` 2명 전용인 현재 `bracket_matches` 스키마로는 애초에 리그전 자체를 저장할 방법이 없음.
- 게다가 조가 A/B뿐 아니라 **C/D까지 4개** — `side` 컬럼 체크 제약(`'A','B','final'`만 허용)에도 걸리고, `BracketView.tsx`가 "A트리—결승—B트리" 2분기 레이아웃으로 하드코딩돼 있어서 4개 조를 화면에 그리려면 렌더링 전면 재설계가 필요함을 확인. (처음엔 "체크 제약만 늘리면 될 것"이라고 잘못 판단했다가, 코드 확인 후 정정함)
- 최종적으로 코드 변경 없이 `division`을 `"여자 청년부 개인전 (A·B풀)"` / `"(C·D풀)"` / `"결선"` 3개로 쪼개서, 각 division 안에서는 기존 A/B/final 2분기 구조를 그대로 재사용하는 방식으로 우회.

### 넣은 내용
- `(A·B풀)`, `(C·D풀)` 각각: side A 7매치(1~4라운드1, 5~6라운드2, 7라운드3) + side B 7매치 + final 1매치 = 15개 × 2풀 = 30개
- `결선`: final 1매치 (AB풀 우승자 vs CD풀 우승자, 그랜드파이널)
- 모든 행은 `player1_name`/`player2_name`/`winner_slot`이 전부 **비어있는 뼈대**(TBD) 상태
- 대신 각 매치의 `notes` 컬럼에 원본 사진의 조 편성을 그대로 텍스트로 남겨둠 (예: "1조: #1 이수빈(일산검도교실), #2 김하은(완산제검도관), #3 박수빈(이화검우회) 승자 vs 2조: ... 승자") — 리그전 승자가 정해지면 이 메모를 보고 어느 매치에 이름을 채워야 할지 바로 알 수 있게 하기 위함

## 배포 전 필수 작업

### 1. 실제 리그전/토너먼트 결과 입력 (미완료, 사용자 소관)
`/admin` → 대회 관리 → 사회인대회 펼치기 → 대진표 매치에서, 3개 division별로 각 매치를 열어 `notes`에 적힌 조 편성을 참고해 실제 승자 이름을 채워 넣어야 함. 리그전(3인조) 결과 자체는 시스템에 구조적으로 저장 안 되고, 그 결과로 나온 "조 대표"만 매치의 player로 입력됨.

### 2. 코드 쪽은 배포 걸림돌 없음
이번 세션 코드 변경(`66983af`, `c96e736`)은 이미 push 완료, SQL 마이그레이션 파일도 이미 DB 반영된 내용을 문서화한 것뿐이라 추가로 실행할 SQL 없음.

## 기능 사용 흐름

### 대회 목록 경량 조회 (`/api/competitions?basic=1`)
- 영상 등록 폼처럼 대회 `id`/`name`/`year`만 필요한 화면에서 사용 — venue/참가자/파일/대진표 조인 없이 즉시 응답
- 전체 데이터가 필요한 관리자 화면은 기존 `/api/competitions` (파라미터 없음) 그대로 사용

### 다중 조(A/B/C/D 이상) 대진표를 표현해야 할 때
- 스키마를 건드리지 말고, `division`을 풀(pool) 단위로 쪼개서 각 division 안에서 기존 A/B/final 2분기 구조를 재사용할 것. 대진표 탭은 division이 여러 개면 자동으로 탭 버튼이 생기므로 화면 쪽 추가 작업도 불필요.

## 알려진 제약 / 유의사항

- **3인 리그전(그룹 스테이지) 자체는 구조화된 데이터로 저장되지 않음** — 조 편성 정보는 `bracket_matches.notes`에 자유 텍스트로만 남아있고, 리그전 개별 경기 결과(누가 누구를 이겼는지)는 시스템 어디에도 없음. 이런 3인 리그전 포맷이 앞으로도 반복될 대회라면, 정식으로 그룹 스테이지 테이블을 신설하는 기능 추가를 검토할 것 (지난 세션에 "옵션 C"로 논의했던 것과 동일한 범위 — 대진표 기능 신규 추가(`9e83c1d`)와 비슷한 규모의 작업이 될 것으로 예상).
- **`side` 체크 제약(`'A','B','final'`)은 그대로 둠** — 2개 조를 초과하는 대진표가 필요할 때마다 `side` enum을 늘리려 하지 말고, 이번처럼 `division`을 풀 단위로 쪼개는 패턴을 우선 고려할 것. `BracketView.tsx`가 2분기 레이아웃으로 하드코딩돼 있어서 `side`를 늘리는 건 사실상 렌더링 전면 재설계와 맞먹는 작업임.
- **이번 대진표 데이터는 관리자 UI를 거치지 않고 스크립트로 직접 DB에 삽입됨** — `/api/competitions/[id]/bracket` POST가 하는 것과 동일한 필드 구조로 넣었지만, 실제 API 라우트를 호출한 게 아니라 Supabase REST에 직접 insert한 것이므로 향후 이 31개 행을 수정할 땐 평소처럼 관리자 화면에서 편집하면 됨(특별히 다르게 취급할 필요 없음).
